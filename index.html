How it started

The short version

Like a lot of people who end up here, it started with curiosity more than a plan — opening up a code editor, breaking things, and slowly figuring out why they broke. That curiosity turned into a Computer Science degree, and somewhere along the way "how does this work" became "how do I build this myself."

The web pulled me in first — the instant feedback loop of changing something and seeing it happen. But I quickly cared just as much about what was happening behind the interface: how data moved, how an API was structured, why a query was slow and how to make it fast. That curiosity is what pushed me from just front-end work into owning the full stack, from Golang services and ConnectRPC contracts down to the database schema.

AI/ML came in from the same place — not as a separate track, but as another layer of the same question: how do you make software actually understand what a person wants, instead of just reacting to clicks? That's the thread running through the projects I build on my own — a chatbot that parses "spent 15 dollars on coffee yesterday" into a real expense, or a recommendation engine that maps skills to career paths.

My internship at Cross Atlantic Software was the first time that thread got tested against a real deadline — building a multi-agent RAG chatbot from a blank repo to a deployed, working pipeline. It's one thing to get an idea working on your own machine; it's another to ship it, watch it handle real queries, and fix what breaks. That gap between "it works" and "it's shipped" is where I've learned the most.

Principles

How I Work

Own the full stack
I don't like handing off the parts I don't fully understand. Database schema, API contract, UI state — I want to reason about all of it.
Make it actually smart
Shipping a feature isn't the finish line if it could understand the user better. AI/ML is a tool for that, not a buzzword to bolt on.
Sweat the edge cases
Error states, migrations, the message that doesn't parse cleanly — the parts nobody notices when they're handled right.
Keep learning in public
Most of what I know came from building things that half-worked, then figuring out why. I'd rather ship and iterate than wait for perfect.
